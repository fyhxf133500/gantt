# Agent Notes

## 经验教训：编码问题
- 原因：我用 PowerShell 的 `Set-Content` 默认编码（UTF-16LE）重写了 `GanttChart.tsx`，但 Vite/浏览器按 UTF-8 解析，导致中文显示为乱码。
- 规避：后续凡是用 PowerShell 写入源码文件，必须显式指定 `-Encoding utf8`；或优先使用 WSL 内的 `cat > file` 等方式写入，确保 UTF-8。
- 补充：PowerShell 的 `-Encoding utf8` 在 Windows PowerShell 下会写入 UTF-8 BOM，部分构建/热更新链路会出现“模块不提供导出”的报错；写源码尽量用 WSL 写入或写完去除 BOM。
- 校验：若出现中文乱码或导出异常，优先检查文件编码与 BOM；必要时重新以 UTF-8 无 BOM 方式写回。

## 经验教训：Get-Content 误读 UTF-8 无 BOM
- 原因：在 Windows PowerShell 中用 `Get-Content -Raw` 读取 UTF-8 无 BOM 文件时，会按默认编码（通常是 ANSI）解码，导致中文变成乱码；再写回时就把乱码永久保存，触发语法错误（如字符串引号被破坏）。
- 规避：避免用 `Get-Content` 直接读写含中文的 UTF-8 无 BOM 文件；改用 `[System.IO.File]::ReadAllText(path, utf8)` / `WriteAllText`，或在 WSL 里编辑。
- 校验：修改后若出现“Unexpected token”之类解析错误，优先检查源码中中文字符串是否被破坏。

## 经验教训：Vite 端口漂移导致刷新进入错误页
- 原因：开发服务器启动时 `5173` 被占用，而 Vite 默认会自动切到下一个可用端口（如 `5174`）；浏览器标签页仍停留在 `http://localhost:5173/`，刷新后就会落入 `chrome-error://chromewebdata/`，控制台显示跨源加载 `5173` 失败。
- 影响：用户看到的是空白或 Chrome 错误页，容易误判成前端代码崩溃，但真实问题其实是服务地址变了。
- 修复：将 `vite.config.ts` 固定为 `host: "0.0.0.0"`, `port: 5173`, `strictPort: true`，避免开发服务器悄悄漂移到其他端口；当前会话里同步重启到 `5173`。
- 规避：以后凡是“打开项目”或重启开发服务器，都先检查 `5173` 是否监听，再确认浏览器地址与实际服务端口一致；若端口被占用，优先释放旧进程或明确告知用户，而不是接受自动跳端口。

## 经验教训：依赖箭头滚动裁剪边界错误
- 原因：依赖箭头覆盖层按整个甘特容器定位和裁剪，而该容器同时包含左侧任务列表和右侧图表；横向滚动后，依赖线超出图表视口但仍落在总容器内，导致箭头显示到任务列表区域。
- 影响：用户横向滚动甘特图时，依赖箭头会跑到左侧任务列表上，造成连线错位和视觉干扰。
- 修复：将依赖箭头覆盖层的定位、尺寸和坐标换算全部绑定到右侧图表 SVG 视口，使用 `chartSvg.getBoundingClientRect()` 作为路径坐标原点和裁剪边界。
- 规避：后续实现 SVG/Canvas 覆盖层时，必须先确认“坐标原点”和“裁剪视口”是同一个可视区域，不能混用外层布局容器和内部滚动图表。

## 经验教训：依赖箭头修复时误把完整 SVG 当成可视视口
- 原因：`gantt-task-react` 的任务条 SVG 是完整可滚动画布，`chartSvg.getBoundingClientRect()` 返回的是整张画布而不是屏幕上可见的图表窗口；横向滚动后，覆盖层会跟着完整画布偏移并保留超宽尺寸。
- 影响：第一次修复虽然减少了部分左侧溢出，但在滚动后仍可能向左侧任务列表或右侧卡片外绘制箭头，表现为“越修越飘”。
- 修复：覆盖层的定位、尺寸和坐标原点改为使用任务条 SVG 的父级滚动视口 `chartSvg.parentElement.getBoundingClientRect()`，并将覆盖层 SVG 自身设为 `overflow: hidden`。
- 规避：处理第三方甘特图库时，要区分“内容画布 SVG”和“可见滚动视口”；滚动裁剪一律绑定 viewport，不能绑定 content SVG。

## 经验教训：第三方甘特图存在多层滚动容器
- 原因：`gantt-task-react` 的结构不是“SVG -> 可视视口”两层，而是“SVG -> 内容容器 -> 横向滚动/裁剪容器”等多层；直接使用 `chartSvg.parentElement` 仍可能拿到宽度等于完整画布的内容容器。
- 影响：第二次修复依然会让依赖箭头在左右滚动后超出可视甘特区域，因为覆盖层仍绑定在过宽的内部容器上。
- 修复：新增真实视口查找逻辑，沿 SVG 祖先节点向上寻找“比 SVG 窄且 overflow 会裁剪”的元素，将覆盖层定位、尺寸和坐标原点统一绑定到该元素。
- 规避：以后排查第三方组件滚动问题时，不要只看父节点层级；必须同时检查节点实际 `getBoundingClientRect()` 宽度和 `overflow` 样式，用“实际裁剪能力”而不是 DOM 距离判断 viewport。

## 经验教训：CPM 不能忽略甘特图实际日期
- 原因：第一次专业 CPM 实现把所有无前置任务都从 0 开始计算，只使用工期和依赖关系，没有把用户当前甘特图中的实际开始日期作为最早开始约束。
- 影响：多个互不相连的父任务组会被错误平移到同一起点，导致日历上更晚结束的任务组没有被标为关键路径。
- 修复：Forward Pass 初始化时使用 `task.start - projectStart` 作为每个任务的最早开始下界，再叠加 FS/SS/FF/lag 依赖约束。
- 规避：在已有排期的甘特图里做 CPM，应明确区分“从零自动排程”和“基于当前计划日期分析关键路径”；本项目属于后者，实际日期必须参与计算。

## 经验教训：CPM 中可见空档不能默认当作 lag
- 原因：任务依赖之间没有显式 lag 时，曾把当前排期里的可见空档自动推断成计划等待时间。
- 影响：原本可以晚一点完成且不影响项目结束的前置任务，会被误判为关键任务，关键路径从“瓶颈分析”偏成“依赖链高亮”。
- 修复：严格 CPM 下，只有显式 `lag` 才参与依赖约束；没有显式 `lag` 的任务间空档默认计入 slack。
- 规避：关键路径语义必须先确认清楚；本项目采用“瓶颈任务/总时差为 0”的 CPM 分析，不应把 UI 上的空白间隔隐式转成业务约束。

## 经验教训：覆盖层不能遮挡交互浮窗
- 原因：依赖箭头和里程碑覆盖层使用较高 `z-index` 叠在甘特图上方，而第三方 tooltip 浮窗没有显式层级，导致箭头视觉上穿过浮窗。
- 影响：用户查看任务详情时，依赖线和箭头会干扰浮窗内容阅读，尤其是关键路径箭头颜色较醒目时。
- 修复：为 `gantt-task-react` 的 tooltip 容器显式提升 `z-index`，让浮窗始终高于依赖覆盖层。
- 规避：新增 SVG/HTML 覆盖层时，要同步梳理“图形层、交互层、弹窗层”的层级顺序；弹窗/tooltip 应始终处在最高视觉层。

## 经验教训：第三方 milestone 会强制按时间点展示
- 原因：`gantt-task-react` 的 `milestone` 类型天然是零工期时间点，内部渲染和传给自定义任务列表的 `end` 会表现为与 `start` 相同。
- 影响：用户在表单里把里程碑结束时间改到多天后，父任务汇总能感知到新结束时间，但任务列表和图形仍像单日点位，造成“保存没生效”的错觉。
- 修复：任务列表和浮窗使用项目原始任务数据展示日期；甘特图中里程碑改由自定义覆盖层按起止时间绘制可变长度长菱形。
- 规避：使用第三方组件的特殊任务类型时，不能假设回传给自定义 slot 的数据等同于业务源数据；显示字段应优先从本项目状态源读取。

## 经验教训：自定义视觉层必须贴合真实交互层
- 原因：里程碑长菱形最初按点位中心和日期估算宽度绘制，没有直接使用第三方甘特图真实可拖拽条的 `getBoundingClientRect()` 左右边界。
- 影响：用户看到的长菱形和实际可拖拽时间条错位，拖动时会感觉“拖的是方框，不是菱形”，视觉反馈和交互反馈不一致。
- 修复：里程碑覆盖层改为直接使用底层可拖拽条的真实 `x/width`，长菱形左右边界与真实拖拽边界一致。
- 规避：凡是覆盖层承担交互提示，坐标必须来自真实交互元素，而不是重新按日期或中心点估算；否则很容易出现“看见的”和“拖到的”不一致。

## 经验教训：弹出菜单不能放进会裁剪的滚动容器
- 原因：Sidebar 项目“更多”菜单渲染在 `.project-list` 内，而项目列表使用 `overflow: auto`；菜单向下展开时被滚动容器裁剪，用户必须滚动很小的列表区域才能看到“重命名/删除项目”。
- 影响：弹出菜单看起来像被挡住或可显示区域过小，项目管理操作不可直接访问，交互体验明显错误。
- 修复：菜单改为按触发按钮坐标计算位置，并使用 `position: fixed` 渲染为 viewport 级浮层，避免被 Sidebar 列表的 overflow 裁切。
- 规避：以后实现 dropdown/context menu/popover 时，先检查祖先节点是否有 `overflow: hidden/auto/scroll`、`transform` 或新的 stacking context；操作菜单优先做成 portal/fixed 浮层，不能默认塞在滚动列表项内部。

## 工作约定：所有 bug 修复都要沉淀经验教训
- 要求：以后每修复一个 bug，都必须在本文件追加一条“经验教训”。
- 格式：至少包含“原因、影响、修复、规避”四项，必要时补充“校验”。
- 目标：避免同类问题反复出现，让布局、数据、第三方组件、存储迁移等坑点逐步变成项目知识库。

## 经验教训：第三方甘特表头会异步重绘
- 原因：日视图双行表头最初只在一次 `requestAnimationFrame` 后处理第三方库生成的 SVG 文本，但 `gantt-task-react` 会在内部 state、日期范围或 DOM mutation 后重新生成表头，导致自定义的 `tspan` 拆行被覆盖。
- 影响：用户切换或刷新后，日视图表头又回到单行 `周三, 11` 格式，日期文字拥挤，看起来像功能失效。
- 修复：对甘特图根节点增加受控的 `MutationObserver` 和短延迟重试，表头文本重绘后再次格式化；同时用 `data-gantt-day-header` 标记已处理文本，避免重复处理。
- 规避：对第三方组件做 DOM 后处理时，不能只处理首次渲染；必须考虑组件内部异步重绘、虚拟 DOM 替换和 mutation 覆盖，必要时做幂等 formatter。

## 经验教训：不要用 padding 制造不改变布局的安全区
- 原因：为了让 Main 右侧露出的安全区域变白，曾在 `.project-workspace` 上使用 `padding-right: 24px`；由于项目全局使用 `box-sizing: border-box`，这个 padding 会从内容盒里扣除宽度。
- 影响：甘特图容器虽然仍写着 `width: 100%`，但实际可用宽度已经被右侧 padding 吃掉，表现为甘特图区域变窄，时间轴和图表内容没有铺满 Main。
- 修复：移除 `.project-workspace` 的右侧 padding，改为让 `.app-main` 和 `.project-workspace` 背景都使用白色；视觉上的右侧白区不再通过布局 padding 实现。
- 规避：以后若目标是“只改变背景/视觉缓冲，不改变内容宽度”，优先调整父级背景、伪元素或非布局型视觉层；不要在承载图表、表格、画布的关键容器上加 padding/margin 来模拟安全区。

## 经验教训：甘特图修复必须覆盖多项目和多视图
- 原因：时间轴宽度问题最初只在默认项目的日视图下观察，默认项目任务跨度较长，掩盖了短项目在日/周/月视图下由第三方库按任务范围生成较窄画布的问题。
- 影响：默认项目看起来恢复正常，但其他项目和周/月视图仍然只生成较短时间轴，右侧大面积空白，用户切换项目或视图后问题再次出现。
- 修复：在 `GanttChart` 内按当前容器宽度、当前视图和当前项目日期范围动态补齐时间轴范围；通过透明范围扩展任务只影响第三方库的 date range，不进入任务列表、业务任务、依赖、关键路径和 localStorage。
- 规避：凡是修复甘特图布局、时间轴、滚动、覆盖层问题，必须至少检查“长项目/短项目/空项目”和“日/周/月”三类视图组合，不能只用默认项目的单一视图判断修复成功。

## 经验教训：时间轴补齐不能吃掉视觉安全区
- 原因：为了解决短项目时间轴不满宽，按甘特容器完整可视宽度计算透明范围扩展任务，结果第三方时间轴被补齐到最右边，覆盖了原本想保留的右侧视觉安全区。
- 影响：布局宽度恢复了，但右侧留白又消失，任务条和时间轴重新贴近屏幕右边缘。
- 修复：把右侧安全宽度纳入时间轴范围计算，从 `visibleTimelineWidth` 中扣除 `TIMELINE_RIGHT_SAFE_WIDTH`，让补齐后的时间轴内容天然停在安全区左侧；不再用容器 padding 挤压甘特图宽度。
- 规避：处理图表/画布“满宽”和“留白”时，要区分容器宽度、内容生成范围、视觉安全区三件事；安全区应参与内容范围计算，而不是事后用布局 padding 或盲目补满。
- 补充：仅扣除 24px 不一定可见，因为时间轴按整列生成，`ceil(width / columnWidth)` 可能仍得到相同列数；应至少预留一个时间单位宽度，并用 `floor` 计算可见单位，避免安全区被整列取整抵消。
- 再补充：第三方甘特图的网格、行背景和 SVG 内容会绘制到可视容器边界；仅调整日期范围或滚动范围不能稳定制造“视觉留白”。如果目标是视觉安全区且不能改变布局宽度，应使用不参与布局的遮罩/视觉层（例如 `::after` 白色覆盖层），并设置 `pointer-events: none` 避免影响拖拽和滚动。

## 经验教训：DOM 后处理标记不能代替真实结构校验
- 原因：日视图双行表头用 `data-gantt-day-header` 标记已处理节点，但 `gantt-task-react` 可能复用同一个 SVG `text` 节点并把内容重新写回 `周一, 16`；旧标记仍保留，格式化逻辑误判为已处理。
- 影响：少数日期表头会偶发退回单行格式，刷新后因为 DOM 重新创建又恢复，表现为不稳定、难复现的视觉 bug。
- 修复：幂等判断改为同时检查标记和实际 `tspan.calendar-day-weekday` / `tspan.calendar-day-date` 结构；如果结构缺失，即使标记存在也重新拆成双行，并监听 `characterData` 变化。
- 规避：对第三方组件做 DOM patch 时，`data-*` 只能作为辅助缓存，不能作为唯一事实来源；必须校验真实 DOM 结构是否仍符合预期，尤其要考虑节点复用、文本重写和异步 mutation。

## 经验教训：页面上下文标题只能有一个主展示位置
- 原因：全局 Main Header 已展示当前项目名，甘特图工具栏又新增了“当前项目 / 项目名”，两个区域承担了同一个上下文说明职责。
- 影响：界面出现两个“默认项目 / 项目 2”标题，用户会感觉信息重复、层级混乱，也让工具栏从操作区变成了混合标题区。
- 修复：保留 Main Header 作为当前项目的唯一主展示位置，移除甘特图工具栏里的项目名和对应无用 prop，让工具栏只承载视图切换、关键路径和新建任务。
- 规避：新增 App Shell 后，要先划分“全局上下文、局部工具栏、内容标题”的职责；同一语义信息不要跨多个层级重复展示，除非有明确的导航或面包屑用途。

## 经验教训：第三方 SVG 表头高度要按真实基线计算
- 原因：`gantt-task-react` 的月份行基线是 `headerHeight * 0.45`，周/日期行基线是 `headerHeight * 0.8`；在 50px 表头内再把日视图拆成双行，会让自定义星期行靠近月份行。
- 影响：日视图出现“三月”和“周一”重叠，周视图出现“三月, 2026”和 `W12` 贴近甚至重叠，表头可读性下降。
- 修复：将甘特图表头高度提升到 64px，并把日视图自定义双行文字整体放到表头下半区，给月份行和日期/周标签行留出稳定间距。
- 规避：调整第三方 SVG 表头时，不能只看外观截图或单个 `y` 值；要先确认上层/下层文字基线公式，再按字体高度和两行文本需求预留真实垂直空间。

## 经验教训：第三方组件的内部滚动缓存要按数据域重置
- 原因：`gantt-task-react` 会缓存 `currentViewDate` 和横向 `scrollX`；切换回曾经打开过的项目时，`viewDate` 值可能与缓存一致，库就不会重新定位，但 `scrollX` 仍保留上一个项目的横向位置。
- 影响：项目 2 的任务日期是 4 月 26-29，切换后时间轴却停在 5 月中旬；刷新页面或切换视图会重置/改变内部缓存，所以看起来又自动修复。
- 修复：将当前 `activeProjectId` 传入 `GanttChart`，并作为第三方 `Gantt` 组件的 `key`，让项目切换时重建甘特内部滚动状态，重新按当前项目的 `viewDate` 定位。
- 规避：多项目/多数据集复用第三方复杂组件时，必须明确“数据域边界”；如果组件内部有滚动、选中、日期定位等缓存，切换数据域时要用稳定 key 或显式重置机制隔离状态。

## 经验教训：筛选结果不能绕过树节点展开状态
- 原因：任务筛选为了满足“父任务匹配时显示子任务”，直接把匹配父任务的所有后代放进扁平结果，并把父任务强制设为展开。
- 影响：在“父任务”筛选下点击小三角不会折叠子任务，因为筛选结果每次都会重新把子任务展示出来，用户感觉折叠功能失效。
- 修复：筛选阶段仍计算命中集合和上下文关系，但最终显示时额外判断祖先节点是否已折叠；如果命中任务的父级处于折叠状态，则隐藏该后代，展开后再显示。
- 规避：搜索/筛选、排序、分组等视图层功能不能绕过树组件的交互状态；生成显示列表时必须把“命中规则”和“展开/折叠可见性规则”分开处理。

## 经验教训：输入框自动选中只能在进入编辑态时执行
- 原因：项目重命名输入框的自动 `focus()` / `select()` effect 依赖整个 `editingProject` 对象，用户每输入一个字符都会更新对象并重新全选文本。
- 影响：中文输入法提交文字后，下一次继续输入会覆盖前面刚输入的内容，看起来像名称被重置，无法连续输入。
- 修复：自动聚焦和选中只依赖 `editingProjectId`，仅在进入某个项目的重命名状态时执行一次；同时在输入法组合态下忽略 Enter/Esc 快捷键，避免误保存或取消。
- 规避：受控输入框内的自动选择、滚动、格式化等副作用不能依赖会随输入变化的完整状态对象；应绑定到“进入编辑态”的稳定标识，并考虑 IME composition 状态。

## 经验教训：不要混用 localhost 和 127.0.0.1 访问本地项目
- 原因：浏览器 localStorage 按 origin 隔离，`http://localhost:5173/` 和 `http://127.0.0.1:5173/` 是两个不同 origin；为绕过 Windows 代理问题临时切到 `127.0.0.1` 会看到另一份空的 localStorage。
- 影响：项目服务正常，但页面只显示“默认项目”，用户会误以为昨天创建的项目数据丢失；实际数据仍保存在 `localhost:5173` 对应的 localStorage 下。
- 修复：确认 `http://localhost:5173/` 打开后原数据仍在，说明不是数据清空，而是访问域名切换导致读取了另一份存储。
- 规避：以后启动本项目固定使用 `http://localhost:5173/` 作为主入口；若 localhost 因代理打不开，优先处理代理绕过规则或强制刷新，不直接切换到 `127.0.0.1`。若必须临时切换地址，需要明确提示 localStorage 不共享，必要时再做数据迁移。

## 经验教训：任务状态判断不能只看当前日期
- 原因：计划状态最初按“当前日期 < start”直接判定为“未开始”，没有先检查任务进度；但用户可能提前开工，进度已经大于 0。
- 影响：任务实际已经开始推进，却在列表中显示“未开始”，状态语义和业务事实冲突，容易误导项目进展判断。
- 修复：调整 `scheduleStatus` 派生顺序：`progress >= 100` 先判定为已完成；超过结束日期且未完成判定为已延期；只要 `progress > 0` 即判定为进行中；最后才根据当前日期早于开始时间显示未开始。
- 规避：派生业务状态时要先识别“事实信号”（如进度、人工确认），再使用“计划日期”推断状态；日期只能作为计划约束，不能覆盖已经发生的真实进度。

## 经验教训：健康概览统计必须先定义清楚计数口径
- 原因：项目健康概览最初同时把普通任务、milestone 节点和父任务派生状态混在同一组统计里；例如“已延期”包含了 milestone 延期节点，而“待确认节点”又排除了延期节点，导致界面数字和列表状态对不上。
- 影响：用户看到的总任务数、待确认节点数和已延期数量无法从任务列表中直观核对，容易误以为父任务被计入或数据统计错误。
- 修复：统一统计口径：summary task 永远不参与健康概览计数；普通任务按 `scheduleStatus` 统计进行中/已完成/已延期；milestone 只按节点确认状态统计待确认/已通过；总任务数统计所有非 summary 工作项（普通任务 + milestone 节点）。
- 规避：新增任何聚合指标前，必须先明确“统计对象集合”和“状态维度”两件事；不要把派生展示状态、人工确认状态和父任务汇总状态放进同一个计数口径里。

## 经验教训：依赖异常只能提示，不能篡改用户事实
- 原因：任务进度可能由用户手动录入，虽然它可能违反 FS/SS/FF 依赖关系，但直接把进度回滚或自动改状态会覆盖用户输入的真实业务信息。
- 影响：如果系统为了满足依赖而自动把进度改回 0 或改写完成状态，用户会丢失已录入进展，也无法区分“数据事实”和“计划规则冲突”。
- 修复：依赖校验改为派生状态，只计算 `dependencyBlocked/dependencyViolation` 并在状态列提示“依赖未满足 / 依赖异常”；milestone 确认通过时若依赖未满足，仅弹出风险提示并默认阻止，用户明确选择后才允许强制通过。
- 规避：状态校验类功能应遵循“事实字段不自动改、规则冲突显式提示”的原则；除非用户明确选择自动修复，否则不要用校验逻辑反向覆盖进度、日期或人工确认状态。

## 经验教训：关键确认不要使用浏览器原生 confirm
- 原因：milestone 依赖未满足时使用 `window.confirm` 做强制通过确认，原生弹窗位置和样式由浏览器控制，无法与项目 UI 对齐，也会出现在页面偏上的位置。
- 影响：弹窗视觉上脱离应用布局，显得突兀且不够专业；用户反馈“太靠上”，也无法展示结构化依赖风险详情。
- 修复：移除原生 `confirm`，改为项目内居中的自定义确认弹窗，列出依赖类型、前置任务和冲突说明，并提供“取消 / 强制通过”按钮。
- 规避：涉及数据风险、删除、强制通过等关键确认时，优先使用项目内 Dialog 组件；浏览器原生 confirm/alert 只适合作为临时调试或极简兜底，不应进入正式交互。

## 经验教训：UI 预判必须和 hook 拦截条件一致
- 原因：milestone “确认通过”按钮只根据当前任务进度计算依赖异常；未通过的 milestone 当前 `progress = 0`，所以 UI 认为没有异常并直接调用 hook。但 hook 会用“通过后的 progress = 100”重新校验依赖并静默拦截。
- 影响：用户点击“确认通过”没有弹窗、也没有状态变化，看起来像按钮失效，实际是 UI 预判和状态更新层的拦截条件不一致。
- 修复：状态列点击“确认通过”时改为按 `assumeProgress: 100` 预判依赖风险；如果通过后的状态会违反 FS/SS/FF，就先弹出项目内确认框，用户选择强制通过后再调用 hook 的 `force` 路径。
- 规避：任何会被 hook/service 拦截的交互，都要在 UI 层用同一套条件提前给出反馈；不能让底层静默返回成为用户可见的“没反应”。
